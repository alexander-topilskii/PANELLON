# Открытые вопросы и вариации

Вопросы, требующие решения перед или в процессе реализации.

---

## 1. Перебор символов — реализуемость

**Проблема:** Полный перебор даёт экспоненциальный рост. 50 символов, длина 20 → 50²⁰ вариантов. Даже 10⁶ проверок по 1 мс = ~17 минут.

**Варианты:**
- [ ] Перебор **по длине** (сначала 1 символ, потом 2, 3…)?
- [ ] Есть ли **грамматика** — только валидные GLSL-выражения?
- [ ] Комната = **N-й успешный результат** при фиксированном порядке перебора (Room_ID задаёт N)?
- [ ] Другой подход (ограничение алфавита, шаблоны, библиотека примитивов)?

## 2. Связь «сетка» ↔ «перебор»

Как позиция в сетке (floor, x, z) связана с генерируемым шейдером?

**Варианты:**
- [ ] **(floor, x, z)** → hash → seed → **N-й успешный** шейдер в детерминированном переборе
- [ ] **(floor, x, z)** → hash → **прямая генерация** строки по правилу (без полного перебора)
- [ ] Другая схема?

## 3. Перевёрнутая пирамида — геометрия

**Вопрос:** Основание вверху или внизу?

**Варианты:**
- [ ] Основание **вверху** — этаж 1 узкий, этаж 500+ широкий
- [ ] Основание **внизу** — этаж 1 широкий, этаж 500+ узкий

**Вопрос:** Как меняется количество комнат по этажам?
- [ ] Формула (линейная, квадратичная)?
- [ ] Фиксированные значения для зон (1–5, 6–50, 500+)?
- [ ] Этаж 500+ — одна комната или много?

## 4. «Дверь не рисуется» — когда определяем?

Как система знает, что в ячейке «невозможный» шейдер?

**Варианты:**
- [ ] **Заранее** — проверяем при генерации этажа, кэшируем «пусто»
- [ ] **На лету** — при приближении к ячейке, возможны задержки
- [ ] Гибрид?
