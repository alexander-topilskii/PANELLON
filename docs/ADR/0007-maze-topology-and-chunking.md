# ADR 0007: Eager Maze Topology + Chunked Mesh Rendering

- Status: Accepted
- Date: 2026-02-23
- Supersedes: clarifies ADR-0003

## Context

TECH_SPEC §3.1 описывает DFS-maze на сетке N×N (spanning tree, гарантированная связность). TECH_SPEC §4 и ADR-0003 требуют chunked lazy generation для больших этажей. Между ними существует противоречие: DFS требует обхода **всех** ячеек для построения spanning tree, а chunking подразумевает, что этаж не генерируется целиком.

Нужно определить, **что именно** генерируется лениво, а что — целиком.

## Analysis

### Стоимость полной DFS-топологии

Топология лабиринта — это компактная структура: для каждой ячейки хранятся 4 бита (наличие стены N/S/E/W). Память и время по этажам:

| Этаж | rooms | side | Ячеек (side²) | Память (topology) | Время DFS |
|------|-------|------|---------------|--------------------|-----------|
| 10   | 2     | 2    | 4             | < 1 KB             | < 1 ms    |
| 100  | 101   | 11   | 121           | < 1 KB             | < 1 ms    |
| 500  | 2501  | 51   | 2601          | ~1.3 KB            | < 1 ms    |
| 1000 | 10001 | 101  | 10201         | ~5 KB              | < 5 ms    |
| 5000 | 250001| 500  | 250000        | ~125 KB            | ~50 ms    |
| 9999 | 999801| 1000 | 1000000       | ~500 KB            | ~200 ms   |

Реалистичный диапазон игры — этажи 0–500 (до 1000 для очень долгих сессий). Дойти до этажа 9999 пошагово невозможно за разумное время (~14 часов непрерывных переходов).

### Стоимость мешей коридоров

Каждая ячейка = до 8 стеновых сегментов + пол + потолок. При ~10 draw calls на ячейку и 1000 ячеек в активной зоне — это ~10K draw calls, что уже на границе бюджета. Mesh generation — CPU-bound, ~1 мс на ячейку.

### Вывод

Узкое место — **mesh generation**, не topology. Maze topology дёшева даже для экстремальных этажей. Chunk-local maze алгоритмы (без полного DFS) существенно усложняют реализацию и не дают гарантии связности без дополнительных швов.

## Decision

Разделяем генерацию этажа на два уровня:

### 1. Maze topology — eager (полная генерация)

При входе на этаж:

1. Вычислить `side = ceil(sqrt(floor(1 + 0.01 * floor²)))`.
2. Выполнить iterative DFS на сетке `side × side` (TECH_SPEC §3.1).
3. Результат — компактная bitfield-структура (`Uint8Array`, 1 байт на ячейку: 4 бита стен + 1 бит visited).
4. Хранить в runtime до перехода на другой этаж, затем dispose.

Гарантии:
- Полная связность (spanning tree).
- Детерминизм: `mazeSeed = hash(globalSeed, floor)`.
- Время: < 5 мс для этажей до 1000, < 200 мс для этажа 9999.
- Память: < 10 KB для реалистичных этажей, < 1 MB для экстремальных.

### 2. Corridor meshes — chunked (ленивая генерация)

После построения topology:

1. Определить active radius (20 ячеек от игрока).
2. Для каждой ячейки в радиусе: прочитать стены из готовой topology → создать mesh.
3. При движении: добавить новые ячейки, dispose дальних.
4. На этажах ≤ 20×20: сгенерировать все меши сразу.

Chunk — группа мешей, управляемая как единица для add/remove. Chunk size = N×N ячеек (default N=16, из TECH_SPEC §4). Chunk не влияет на topology — только на mesh lifecycle.

### Стратегия для экстремальных этажей (side > 500)

Если DFS занимает > 100 мс (измеряем runtime):
- Выполнять DFS асинхронно (через `requestIdleCallback` или batched с yield).
- Показывать loading state во время fade-transition (бюджет 700 мс, PERFORMANCE_BUDGET).

## Consequences

Плюсы:

- Стандартный DFS без модификаций — простая и надёжная реализация.
- Гарантированная связность на любом этаже.
- Chunk boundary не требует согласования топологии — всё уже готово.
- Maze topology переиспользуется для размещения лестниц, дверей, preview.

Минусы:

- Для этажей > 5000 DFS может занять > 50 мс. Mitigation: async batching.
- Память topology на экстремальных этажах — до 1 MB. Приемлемо, учитывая что в runtime один этаж.
